# Story 3-4: AI 章节级方案生成 — UX 设计规格

## 概述

本设计规格定义 AI 章节级方案生成功能在 `proposal-writing` 阶段的交互行为，覆盖从用户触发生成到内容插入编辑器的完整流程。设计基于 BidWise UX 规范中的“异步操作 + Toast 回调”“内联错误恢复”“章节级独立迭代”三条模式。

**范围约束：**

- Story 3.4 交付的是“章节生成、重新生成、进度、排队、错误恢复、任务恢复”
- 右侧 `AnnotationPanel` 在本 Story 仅允许显示轻量级生成摘要或占位文案
- Screen 3 PNG 中的 AI 建议卡 / 评分预警卡、以及“AI 生成 · 待来源标注”标签属于后续 Story 3.5 / Epic 4 的未来态预览，不构成 3.4 的必须实现项

## 核心组件设计

### 1. ChapterGenerateButton — AI 生成按钮

**触发条件：**

- 章节正文为空，或
- 章节正文仅包含 Story 3.3 写入的 guidance blockquote / 空段落

**视觉设计：**

- 位置：章节标题（H1-H4）右侧，与标题同行
- 样式：Ant Design Ghost Button，蓝色边框 + 蓝色文字
- 图标：`<ThunderboltOutlined />` + 文字“AI 生成”
- 尺寸：`size="small"`
- 悬浮态：蓝色填充 + 白色文字，200ms `ease-out`

**交互：**

- 点击即触发生成，无额外确认
- 点击后按钮立即消失，替换为 `ChapterGenerationProgress`
- 对于“只含 guidance”章节，生成后应替换原 guidance blockquote，而不是追加在其后

### 2. ChapterGenerationProgress — 生成进度

**视觉设计：**

- 占据章节正文区域，替代 guidance/空白内容
- 骨架屏：3-8 行灰色脉冲条
- 阶段进度文字：骨架屏上方，蓝色文字 + 旋转图标
- 淡入动效：200ms `ease-out`

**阶段进度显示：**

| 条件 | 阶段标识 | 显示文字 | 说明 |
|------|---------|---------|------|
| `task.status === 'pending'` | `queued` | `排队中...` | 超过 `maxConcurrency=3` 时显示 |
| 0-24% | `analyzing` | `正在解析章节上下文...` | 读取当前 proposal/requirements/mandatory/seed |
| 25-49% | `matching-assets` | `正在匹配相关资产...` | Alpha 占位，快速跳过 |
| 50-89% | `generating` | `AI 正在生成章节内容...` | 主耗时阶段 |
| 90-99% | `annotating-sources` | `正在标注内容来源...` | Alpha 占位，快速跳过 |
| 100% | `completed` | — | 骨架屏淡出，正文淡入 |

**进度条：**

- Ant Design `Progress`
- 细线型、蓝色、无百分比数字

### 3. RegenerateDialog — 重新生成浮层

**触发方式：**

- 鼠标悬浮在任意已有正文的章节标题上，显示 `<RedoOutlined />` 图标按钮
- 不要求预先识别该正文是否由 AI 生成

**浮层设计（Ant Design Modal）：**

```text
┌─────────────────────────────────────────────┐
│  重新生成章节                            [×] │
├─────────────────────────────────────────────┤
│  当前章节：2. 系统总体架构设计               │
│                                             │
│  补充上下文（可选）                          │
│  ┌─────────────────────────────────────┐    │
│  │ 请补充您希望 AI 侧重的方向、额外要   │    │
│  │ 求或上下文信息                       │    │
│  └─────────────────────────────────────┘    │
│                                             │
│  ⚠ 重新生成将替换当前章节内容               │
│                                             │
│              [取消]    [重新生成]            │
└─────────────────────────────────────────────┘
```

**交互细节：**

- Modal 宽度：480px
- TextArea：4 行高度，最大 500 字符
- 确认后关闭 Modal，章节进入“生成中”
- 若系统已检测到“该章节在任务执行期间被人工继续改动”，则不在此 Modal 中直接覆盖，而在完成阶段进入冲突确认

### 4. 冲突确认（非主路径）

**触发条件：**

- AI 任务开始后，目标章节正文发生人工修改
- 任务完成时发现 `baselineDigest` 已变化

**交互原则：**

- 不自动覆盖
- 使用 `Modal.confirm`
- 默认主文案为“该章节在生成期间已被修改，是否用 AI 结果覆盖当前内容？”
- 默认建议动作：保留当前内容

### 5. InlineErrorBar — 内联错误条

**触发条件：**

- `ai-proxy/provider-adapter` 3 次自动重试后仍失败

**视觉设计：**

- 位置：章节标题下方，正文区域内
- 背景：浅红色 `#FFF1F0`
- 左侧：3px 红色竖线
- 圆角：6px
- 内边距：12px 16px
- 图标：`<CloseCircleOutlined />`
- 错误文字：`章节内容生成失败，请选择后续操作`

**三个操作按钮：**

| 按钮 | 样式 | 行为 |
|------|------|------|
| 重试 | 蓝色 Ghost | 重新发起新的 generate 任务 |
| 手动编写 | 默认 Ghost | 关闭错误条，章节恢复正常可编辑状态并聚焦正文 |
| 跳过 | 文字链接 | 关闭错误条，保持该章节当前内容不变 |

### 6. RegenerateButton — 悬浮重新生成按钮

**触发条件：**

- 鼠标悬浮在已有正文的章节标题上

**视觉设计：**

- 位置：章节标题右侧
- 样式：24px 圆形按钮，灰底，`<RedoOutlined />`
- 悬浮显现：200ms 淡入
- Tooltip：`重新生成此章节`

## 多章节并行生成

**场景：** 用户连续点击多个章节的“AI 生成”按钮。

**行为规则：**

- 各章节独立显示 `ChapterGenerationProgress`
- 后端通过 task-queue 并行调度，受 `maxConcurrency=3` 限制
- 超出并发限制的章节显示 `queued` 态文案“排队中...”
- 用户可在任意章节继续编辑，不受其他章节生成阻塞
- 各章节完成时独立触发插入或冲突确认

**文档大纲状态指示：**

- `queued`：橙色时钟图标
- `generating` / `analyzing` / `matching-assets` / `annotating-sources`：蓝色 spinner
- `completed`：绿色 check
- `failed`：红色警示图标

**右侧侧边栏行为：**

- Story 3.4 只显示轻量级摘要：
  - 无任务时：沿用现有占位文案
  - 有任务时：显示“X 个章节正在生成中...”
- 不在本 Story 中实现 AI 建议卡、评分预警卡、来源标签展开等完整批注体系

## 错误恢复流程

遵循 BidWise UX 规范“内联错误恢复”标准模式：

1. **自动重试**：仅在 `ai-proxy/provider-adapter` 层自动重试 3 次（指数退避）
2. **局部失败**：失败只影响当前章节，不影响其他章节和整体编辑
3. **内联展示**：错误在原位展示，不弹全局失败模态
4. **三选一恢复**：重试 / 手动编写 / 跳过
5. **任务恢复**：重新进入同一项目时，应恢复当前 active 章节任务映射

## 动效规范

| 元素 | 触发 | 动效 | 时长 | 缓动 |
|------|------|------|------|------|
| 骨架屏出现 | 点击“AI 生成” | 淡入 | 200ms | ease-out |
| 骨架屏消失 | 生成完成 | 淡出 | 300ms | ease-out |
| 正文出现 | 骨架屏消失后 | 淡入 | 300ms | ease-out |
| 错误条出现 | 生成失败 | 从上滑入 | 200ms | ease-out |
| 悬浮重新生成按钮 | 鼠标悬浮标题 | 淡入 | 200ms | ease-out |
| Modal 弹出 | 点击重新生成/冲突确认 | 缩放 + 淡入 | 300ms | ease-in-out |

## 响应式行为

- `1920px+`：三栏完整显示（大纲 240px + 编辑器 + 批注 320px）
- `1366px`：批注侧边栏可折叠为图标模式
- 编辑器内容区最大宽度：800px
- `ChapterGenerateButton`：编辑器宽度窄于 600px 时仅显示图标

## 无障碍要求

- 所有按钮提供 `aria-label`
- 骨架屏区域提供 `aria-live="polite"`
- `InlineErrorBar` 使用 `role="alert"`
- `RegenerateDialog` 使用 Modal 标准焦点管理
- 大纲状态图标应为屏幕阅读器暴露章节状态标签，如“系统总体架构设计，生成中”
- 键盘可达：Tab 导航到生成按钮 / 重新生成按钮，Enter 触发

## 原型页面规划

本设计包含以下原型画面：

1. **Screen 1 — 空章节态**：章节只含标题或 guidance-only 占位，显示 `AI 生成`
2. **Screen 2 — 生成进度态**：单章节生成中，显示骨架屏 + 四阶段进度
3. **Screen 3 — 生成完成态**：章节正文已填充；右侧卡片为未来态组合参考，不作为 Story 3.4 gate
4. **Screen 4 — 重新生成对话框**：Modal 浮层 + 上下文输入
5. **Screen 5 — 错误态**：`InlineErrorBar` + 三选一操作按钮
6. **Screen 6 — 多章节并行**：多个章节分别处于 generating / analyzing / queued
